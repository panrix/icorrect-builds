# Ferrari Dependency Assessment
Last updated: 2026-04-24 13:49 UTC

## Purpose
Document the current operational dependency on Ferrari, estimate what value is replaceable through stronger process and agent design, and identify what would actually be lost if workshop organisation improved.

## Current State, Reality Today
Ferrari appears to function less as a true control owner and more as a communication and interpretation layer across enquiry, triage, quote, and customer-facing progression work.

Observed current-value areas:
- translating technical findings into customer-facing language
- handling ambiguous customer communication
- carrying context between systems, people, and conversations
- spotting some workflow and product gaps and suggesting fixes
- performing manual admin tasks such as invoicing, chasing, and queue follow-up when these are not system-enforced

At the same time, today’s process capture suggests a meaningful portion of Ferrari’s apparent value exists because the system is under-documented, control-light, and dependent on humans remembering what “good” looks like.

## Current Failure Modes
Current dependency on Ferrari is inflated by process weakness in the following ways:
- undocumented decision logic forces humans to act as memory layers
- weak intake and quote controls create avoidable clarification work later
- customer communication standards are not yet fully codified into SOP and agent behaviour
- invoicing and follow-up controls appear too dependent on manual ownership rather than enforced workflows
- technical nuance lives in people and chats instead of in canonical KB/process documents

This means Ferrari may often be solving problems the system should have prevented.

## Working Hypothesis
Ferrari’s operational value is currently inflated by process gaps.

If the right knowledge base, SOPs, workflow gates, and customer-service agent behaviour were in place, the business would likely lose less than it currently feels like if Ferrari were removed.

## Replaceable Value
Likely replaceable or materially reducible through stronger organisation, SOPs, workflow design, and agent support:
- standard enquiry handling
- standard triage clarification
- repair-type qualification prompts
- standard quote follow-up
- stale-job chasing
- invoicing prompts and invoice control checks
- queue movement reminders
- routine customer status communication
- internal handoff summarisation
- multi-system context stitching where the rules can be made explicit

## Harder-to-Replace Value
Likely harder to replace in the short term:
- nuanced technical-to-customer translation
- edge-case interpretation where symptoms, diagnostics, and quote language do not map cleanly
- judgment in unusual customer situations
- identifying subtle workflow/product issues before they become obvious metrics problems
- relationship continuity in existing customer/corporate threads where trust sits with the individual rather than the system

## What We Would Actually Lose If Organisation Improved
If the organisation, process docs, and agent layer were materially stronger, the likely remaining loss from Ferrari would narrow to:
- specialist exception handling
- technical communication nuance
- some product/process ideation
- some continuity on active relationship threads

The business should lose much less on routine operations than it would today because the system would carry more of the load.

## System Implications
To reduce dependency on Ferrari-like human glue roles, the system needs:
- canonical SOP and KB coverage for enquiry, triage, quote, acceptance, repair readiness, and invoicing controls
- explicit gate logic for movement between stages
- stronger customer-service agent personality and response policy grounded in workshop truth
- enforced invoice-follow-up and corporate billing controls
- structured templates for technical findings translated into customer-safe language
- exception-routing rules so only true edge cases escalate to a specialist human

## Decision Implication
Ferrari should not be treated as either fully disposable or uniquely indispensable.

The more accurate view is:
- some of his value is real
- much of his day-to-day importance appears to come from system weakness
- better organisation should compress the role significantly
- the business should aim to retain knowledge and systemise the repeatable parts, not stay dependent on a person-shaped process gap

## Open Questions
- what percentage of Ferrari’s current workload is routine vs true exception handling?
- how much technical translation can be safely captured into templates and agent behaviour?
- which current customer/corporate relationships are person-dependent vs system-transferable?
- which current failure points would remain even after SOP and workflow improvement?
