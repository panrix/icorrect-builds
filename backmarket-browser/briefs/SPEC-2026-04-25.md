## Scope
This runtime exists for seller-portal work that the BM API does not cover safely enough today. The current implementation should use a VPS-first `browser-use/browser-harness` approach before defaulting to raw Playwright. Ricky reported a Claude Code browser-harness session was materially better than Playwright for Monday automations; the completed pilot evidence lives at `/home/ricky/builds/monday/browser-harness-pilot-report.md` and `/home/ricky/builds/monday/browser-harness-monday-skill.md`.

Important product direction from Ricky: the durable runtime should be installed around and run through the VPS. Local browser-harness work proved the pattern, but the productised version should live on the VPS. The method is: run one tiny canary, for example one SKU correction; confirm it end-to-end; compile the successful path into a reusable skill; then replicate safely from that skill.

The first release has four responsibilities.
- Authenticate to the seller portal and satisfy the email-code step through a local SMTP-forwarded mailbox path.
- Maintain a selector map and validation harness so DOM drift fails closed.
- Execute fix-sku against approved listings only, with checkpoints and rollback data.
- Poll customer service, warranty, and returns tabs in read-only mode and emit normalized queue events.

Out of scope for Task C
- Customer replies
- Refund actions
- Accept or reject return or warranty actions
- Price, quantity, shipping, or publication-state edits
- Any live execution in this task

## Runtime Modes
- design: docs and fixtures only
- read-only: login, navigate, collect selectors, capture queue data, no form submits
- write-approved: only fix-sku may write, and only with explicit CLI flag plus batch approval
The codebase defaults to read-only. A missing flag or missing approval file is a hard stop.

## Architecture
Core components
- lib/session.js: launches or attaches browser-harness from the VPS, loads storage state, verifies whether the current context is still authenticated, and persists a fresh storage state after successful login.
- lib/mailbox-smtp.js: reads inbound BM login emails from a local spool or SMTP sink, filters on sender and subject, extracts the freshest one-time code, and exposes only code metadata plus a redacted audit record.
- lib/selector-map.js: single source of truth for selector entries. Each entry stores page id, selector strategy, selector string, fallback chain, uniqueness expectation, and the portal-client method that depends on it.
- lib/selector-harness.js: runs selector assertions against cached fixtures and against live read-only validation sessions, then writes drift results into browser.db.
- lib/portal-client.js: high-level actions such as openListings, openListingById, readSkuField, saveListing, openCustomerService, openWarranty, and openReturns.
- lib/lock.js: holds data/.runtime.lock for the full operation and refuses concurrent runs.
- lib/db.js: SQLite wrapper for runs, selector validation history, login-code events, fix-sku checkpoints, cs events, and rollback data.
- lib/screenshots.js: names and stores screenshots for preflight, pre-change, post-change, and anomaly cases.

Operations
- operations/fix-sku.js: the only write path in the first release.
- operations/cs-monitor.js: read-only scan of customer service, warranty, and returns queues.

- Listing cards show product title, `SKU:`, appearance badge, `Archive listing`, inventory units, competition, and market.
- Listing detail page shows `Listing detail`, `Duplicate listing`, product title, `product_id:`, `Appearance:`, editable `SKU` input, market tabs (`Alerts`, `Comment`, `Product specifications`, `Similar listings`), and alerts such as `This listing has no stock available`.

A save click alone is not proof.

## Email-Code Login via SMTP
Constraint: SMTP by itself does not let the runtime read a mailbox. To keep the design compatible with the request, the runtime assumes BM login emails are forwarded into a local SMTP sink or spool that Jarvis controls. The browser runtime never logs into the mailbox provider directly.

Login sequence
1. Start a fresh run record and acquire the runtime lock.
2. Load the latest storage state. If the session is valid, skip credential entry and continue.
3. If the login page appears, enter BM_PORTAL_EMAIL and BM_PORTAL_PASSWORD.
4. When the email-code prompt appears, store the prompt timestamp and wait up to BM_CODE_TIMEOUT_SEC for a matching inbound message in BM_CODE_SPOOL_DIR.
5. Filter candidate messages by sender, subject, recipient, and arrival time after step 3.
6. Parse the code using BM_CODE_REGEX, enter it once, and persist the new storage state on success.
7. Record only metadata in SQLite: message id hash, arrival timestamp, code age, and parse result. Do not store raw message bodies long term.
8. If no valid code arrives, fail closed, capture screenshots, and alert the operator.

Mailbox guardrails
- Accept only BM sender domains and the exact recipient jarvis@icorrect.co.uk.
- Reject reused codes and codes older than the current login attempt.
- Prune raw eml files after parsing unless the run is flagged for anomaly retention.
- Never echo the code to stdout, logs, or alert cards.

## Selector-Mapping Harness
Pages that must be mapped before any write path is enabled
- Login page
- Dashboard landing page
- Listings index
- Listing detail edit page
- Customer service queue
- Warranty queue
- Returns queue

Selector manifest contract
- key
- page
- purpose
- primary selector
- fallbacks
- uniqueness rule
- sample assertion
- owning portal-client method

Harness behavior
- Load cached fixtures from test/fixtures and require every selector to resolve uniquely.
- Run a live read-only validation session across at least 3 sample entries per queue page.
- Save a screenshot and a fixture snapshot for every validated page.
- Persist pass or fail results in selector_observations with DOM hash and timestamp.
- Mark any selector as drifted if uniqueness fails, fallback usage changes unexpectedly, or the owning action throws.
- Block all operations when a drifted selector touches login, listing edit, or queue navigation.

## Fix-Sku Operation
Input contract
- CSV or JSON lines generated from the repaired listings-audit.js output
- Required fields: listing_id, current_sku, correct_sku, a_number, chip, ram, ssd, cpu_cores, gpu_cores, colour, grade
- Optional fields: reason, operator, approval_ref

Execution contract
1. Validate the file schema and reject duplicates or conflicting tuples.
2. Acquire the runtime lock and verify selector health.
3. Navigate to the original Listings page URL.
4. Open `Filter` and search by the strongest available identifier: UUID/listing id first if the portal supports it, otherwise exact `current_sku`, otherwise exact title only as a fallback.
5. Apply filters and require exactly one result.
6. Open the listing detail by clicking the underlined listing name/title.
7. Capture pre-change values for SKU, product_id, appearance, quantity, pub_state, and listing title, plus a screenshot.
8. Assert that the live SKU still equals `current_sku`. If not, stop and mark the row stale.
9. Edit only the SKU field.
10. Click the `Save` button that appears below the SKU field.
11. Do not rely on a toast/success message: Back Market may not visibly confirm completion.
12. Navigate back to the original Listings page URL, filter again using the expected `correct_sku` or stable UUID/listing identifier, reopen the listing detail, and re-read the SKU.
13. Mark success only when SKU equals `correct_sku` and product_id, appearance, quantity, and pub_state are unchanged.
14. Write a rollback record with listing_id, old_sku, new_sku, screenshots, and run id.
15. Sleep for BM_RATE_LIMIT_MS before the next listing.

Hard stop conditions
- selector drift or ambiguous selector
- toast error or missing success toast
- live SKU does not match the expected current value
- quantity changes
- pub_state changes
- unexpected modal or captcha
- more than 1 failure in the first 5-listing canary

Rollback path
- scripts/rollback-fix-sku.js --batch-id <id> replays the reverse records in reverse order
- rollback itself is blocked if the live SKU no longer matches the expected post-change value
- rollback requires the same selector-health gate as forward execution

## Customer-Service, Warranty, and Returns Read-Only Monitor
Goal: replace Ferrari manual tab checking with a normalized queue feed, not with automated decisions.

Sources
- Customer service inbox
- Warranty or repair claims queue
- Returns queue

Collected fields
- queue_type
- portal_thread_id or claim id
- order_id
- listing_id if visible
- customer name
- created_at and updated_at
- status badge
- message preview
- tracking value if visible
- screenshot path
- raw DOM hash for dedupe and drift evidence

Behavior
- The monitor logs in read-only, walks each queue, paginates until the configured limit, and normalizes records into cs_events.
- It deduplicates on queue_type plus portal_thread_id plus updated_at.
- It emits alert cards only for new or materially changed entries.
- It never clicks reply, refund, accept, reject, print label, or close-thread actions.
- It captures screenshots for any entry marked urgent, unread, escalated, or anomalous.

Alert classification
- info: new low-risk message or status change
- attention: unread warranty or return thread older than SLA
- critical: selector drift, missing queue access, portal error banner, or queue item that cannot be normalized

- listing_fix_jobs: batch_id, listing_id, current_sku, correct_sku, tuple_hash, approval_ref, status

- cs_events: queue_type, external_id, order_id, listing_id, status, payload_json, first_seen_at, last_seen_at

## Test Plan
Unit tests
- login email parser accepts only valid sender, subject, and fresh codes
- selector manifest parser rejects missing ownership or missing uniqueness rules
- batch validator rejects duplicate listing ids, duplicate target SKUs, and tuple mismatches
- stop-condition logic halts on quantity drift, pub_state drift, or toast failure

Fixture tests
- every required page has a cached HTML fixture and screenshot
- every selector resolves uniquely on the fixture set
- fallback selectors are exercised intentionally, not silently

Read-only browser tests
- session bootstrap with expired storage state
- login code wait and timeout path
- customer service, warranty, and returns queue navigation
- screenshot capture and alert emission on forced selector drift

Write-path tests before production
- synthetic fix-sku rollback drill against a fake page model
- live preflight that opens but does not save the first 5 approved listings
- production canary of 5 listings only after the operator confirms the read-only preflight and rollback drill

## First 5-Listing SKU Batch Protocol
1. Freeze exactly 5 approved listings in the input file and attach an approval reference.
2. Run scripts/validate-selectors.js and require a full green result on login, listings, listing edit, customer service, warranty, and returns pages.
3. Run a read-only preflight over the 5 listings and save pre-change SKU, quantity, pub_state, title, and screenshots.
4. Run a synthetic rollback drill using the same code path against a fake fixture page.
5. Start scripts/run-fix-sku.js --batch-size 5 --approval-ref <ref> in write-approved mode.
6. After each listing save, reload the page and verify SKU changed while quantity and pub_state stayed stable.
7. Stop immediately on the first unexpected state change or on more than 1 operator-visible anomaly.
8. After all 5, rerun the listings audit and compare against the batch manifest.
9. Manually inspect all 5 listings in the portal and confirm they remain active at quantity 1.
10. Hold the next batch for 24 hours and watch sale-detection.js plus portal alerts for regressions.

## Guardrails
- Default mode is read-only.
- The runtime never stores raw login codes in logs or alert payloads.
- Only the SKU field is whitelisted for write access in the first release.
- All queue monitoring is read-only and must never click decision buttons.
- A stale or missing selector map blocks the entire runtime.
- Every write must have pre and post screenshots plus a rollback record.
- Single-operation lock is mandatory.
- Screenshot retention is 90 days, with anomaly archives kept until manually cleared.
