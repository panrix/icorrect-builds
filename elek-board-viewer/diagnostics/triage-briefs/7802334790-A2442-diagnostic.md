## Diagnosis summary

**Two separate problems:**

### Problem 1: Power cycling at 20V/0.552A

0.552A at 20V with cycling falls in the A2442 flow's **0.5-1.5A (partial boot -- stalling)** scenario, specifically the **cycling** path:

> *Step 1: observe ammeter -> if cycling -> boot loop -> check SSD/NAND*

The board is attempting to boot, reaching ~11W, then restarting. This is characteristic of:
- NAND/SSD power fault (a shorted NAND pulling down a rail, causing the PMU to cycle)
- SoC hitting a fault condition during early boot and resetting
- Missing or unstable power rail that the boot sequence requires

### Problem 2: Host Mac cannot see the board in DFU

Saf replaced all 4 CD chips (CD3217 USB-C controllers) -- so the USB-C hardware should be present. But the board doesn't enumerate on the host Mac during DFU. Possible causes:
1. **Power cycling too fast** -- if the board restarts every 2-3 seconds, the USB enumeration never completes. The host sees the device appear and disappear. This is the most likely cause.
2. **DFU port CD chip not working** -- even though Saf replaced all 4, one may have a cold solder joint or be the wrong part
3. **SPI ROM corruption** -- iBoot/firmware corrupted, board can't enter DFU state properly
4. **Wrong DFU port** -- on A2442, DFU port is the **left-rear USB-C port (ATC0)**

**Key insight:** Problem 2 is almost certainly caused by Problem 1. If the board power cycles every few seconds, USB never stabilizes. Fix the cycling first, and DFU connectivity will likely follow.

## Step 2 — NAND/SSD power check (if still cycling after clean)

The A2442 uses separate NAND PMIC(s) feeding the SSD. From our research:
- **P2V5_NAND0_PGOOD** — on UN700/UN800 (NAND PMICs). Should be ~3.7V. From the Rossmann A2442 case (820-02098, same board), this read 0.752V when faulty.
- **P2V5_NAND1_PGOOD** — on C5841/U5700. Should also be ~3.7V.
- **PP1V2_NAND** rails — check for short to ground
- **TPS62180 equivalent** — the NAND buck converter. If this is internally shorted, it dumps 12.6V PPBUS directly onto the NAND rail (most common NAND failure mode per our research)

**With power off, in diode mode:**

1. **Check PP1V2_NAND for short to ground**
   - `PP1V2_NAND diode: ___` (expect high resistance, not shorted)
2. **Check PP2V5_NAND for short to ground**
   - `PP2V5_NAND diode: ___`
3. **Check P2V5_NAND0_PGOOD at UN700/UN800:**
   - `P2V5_NAND0_PGOOD: ___V` (expect ~3.7V when powered)
4. **Check P2V5_NAND1_PGOOD at C5841/U5700:**
   - `P2V5_NAND1_PGOOD: ___V` (expect ~3.7V when powered)
5. **If either is shorted -> thermal camera at 1V injection into PPBUS_G3H:**
   - `Short found on: ___`
   - `Hot component: ___`
   - Upload thermal photo

**From the Rossmann A2442 case study (820-02098, same board):**
- UN500 was internally shorted on PP1V2_NAND1
- NAND0_OCARINA_PGOOD read 0.752V (should be ~3.7V)
- I2C lines abnormally low (~0.5V)
- Fix required replacing ALL four NAND chips from donor (same positions/order)

## Step 3 — Check DFU port specifically

If the board stops cycling (either from cleaning or from fixing the NAND short):

1. **Connect USB-C to USB-C data cable to the LEFT REAR port (ATC0)**
   - This is the DFU port on A2442
2. **Enter DFU mode:** Power on -> immediately hold all four keys (Left Control + Left Option + Right Shift + Power) for 10 seconds -> release all except Power -> hold Power for 10 more seconds
3. **Does the host Mac show a DFU device?**
   - `DFU detected: YES / NO`
4. **If NO -> check the specific CD chip on ATC0:**
   - Is it getting power? Probe the CD chip's core LDO output (~1.05V expected)
   - `CD chip on DFU port: powered / dead`

---

## Step 4 — If DFU connects, attempt restore

1. **Try Revive first** (preserves data if any -- unlikely on a test unit but good practice)
2. **If Revive fails -> Restore** (full wipe)
3. **If Restore fails -> capture the error code**
   - `DFU result: SUCCESS / ERROR ___`
   - Screenshot of Apple Configurator 2 output

Refer to the 4010 research in `research/RESEARCH-FINDINGS.md` for error-specific diagnosis.

---

## Step 5 — If DFU still doesn't connect

Last resorts:
1. **Check SPI ROM (U2010 or equivalent on A2442)**
   - If SPI ROM is corrupted, the board can't enter DFU properly
   - Can be reprogrammed from a donor or blanked for DFU to re-flash it
2. **Check if SOC_FORCE_DFU pad exists on A2442** -- can force DFU entry by grounding this pad
3. **If none of the above works:** the M1 Pro SoC may be damaged from the liquid event. Given 4x CD chips, audio amps, and multiple passives were already damaged, SoC damage is plausible. Board would be BER.
