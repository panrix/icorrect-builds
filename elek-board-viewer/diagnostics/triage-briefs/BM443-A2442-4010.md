## Diagnosis summary

This is a textbook **NAND communication / firmware update failure**. The progression tells the story:

| Stage | What happened | What it means |
|---|---|---|
| Original NAND -> Error 9 | ASR signature check failed during restore | Original NAND has corrupted firmware or data. SoC can talk to it but the data is bad. |
| 1st new SSD -> Error 4010 | Device disconnected during restore | New SSD not communicating reliably. Either wrong SSD model, not blanked, or data-line issue. |
| 2nd new SSD -> 10% then fails | Restore starts but dies during NAND firmware update | SoC can communicate initially but the write operation fails mid-way. This is **operation 58 ("Updating NAND Firmware")** per our research. |

**The 10% failure point is the key clue.** From the 4010 research:
> *"The DFU restore process runs `/usr/bin/nvmefwupdater` at operation 58 ('Updating NAND Firmware'). If there is an FTL version mismatch, cold solder joints on data lines, or incompatible NAND, it fails here."*

The board boots to 20V/2.2A (normal range), the SoC is alive, USB works (DFU connects), and the restore STARTS. Something breaks during the sustained write to NAND.

## Step 2 — Verify SSD compatibility

**This is the most likely cause of the 10% failure.** Wrong NAND model = FTL mismatch = firmware update fails.

**Reply to Ricky with:**

1. **What is the part number on the replacement SSDs?** (read off the chip packages)
   - `SSD part numbers: ___`
2. **Where did they come from?** (donor board model + serial, or purchased)
   - `Source: donor A____ / purchased from ___`
3. **Were the SSDs blanked/erased before installation?**
   - `Blanked: YES / NO / DON'T KNOW`
   - Non-blank NANDs cause FTL version mismatch at the exact 10% point
4. **How many NAND chips does A2442 use? Are ALL of them replaced, or only some?**
   - `Chips replaced: ALL ___ chips / only ___ of ___ chips`
   - From Rossmann case: ALL NANDs must be replaced together (they form a striped array). Partial replacement = guaranteed failure.
5. **Are the NANDs in the same positions as the donor board?**
   - `Positions match donor: YES / NO / DON'T KNOW`

## Step 3 — If SSDs are compatible and blanked: check the write path

If Step 2 confirms correct model, blanked, all positions correct -- the failure is in the data path between SoC and NAND during sustained writes.

1. **Check P2V5_NAND0_PGOOD** — on UN700/UN800 (NAND PMICs)
   - On the A2442 Rossmann case (820-02098, same board), this read 0.752V when it should be ~3.7V
   - `P2V5_NAND0_PGOOD: ___V`
2. **Check P2V5_NAND1_PGOOD** — on C5841/U5700
   - `P2V5_NAND1_PGOOD: ___V` (expect ~3.7V)
3. **Check I2C lines to NAND**
   - Rossmann case: I2C lines at ~0.5V (should be higher)
   - `I2C NAND: ___V`
4. **Check PPBUS_AON** — test points: CN123 (BOTTOM), CN124 (BOTTOM), CR611 (BOTTOM)
   - `PPBUS_AON: ___V` (expect ~12.6V)
5. **Check PP5V_S2_MAIN stability with oscilloscope during DFU restore**
   - The ERROR-CODE-MAP says: *"Check PP5V_S2 stability during restore (monitor with oscilloscope for dips/dropouts)"*
   - PP5V_S2_MAIN is confirmed on U9300 (Viper VR) among others
   - If PP5V_S2 dips during the heavy write at 10%, the NAND firmware update aborts
   - `PP5V_S2 during restore: stable / dips at ___% / other: ___`

---

## Step 4 — Try restore with original NAND (if still available)

Before investing more time in new SSDs:

1. **Put the original NAND back in** (the one that gave Error 9)
2. **Try DFU Revive** (not Restore) -- this reinstalls firmware without erasing
   - If Revive works -> the original NAND's firmware was corrupted, not the hardware
   - `Revive with original NAND: SUCCESS / ERROR ___`
3. **If Revive fails -> try DFU Restore**
   - `Restore with original NAND: SUCCESS / ERROR ___`

**Why this matters:** Error 9 (ASR verification) can be caused by a corrupted restore image or a bad USB cable -- not necessarily a NAND hardware fault. If the original NAND works with a fresh DFU attempt using a verified cable and host, the whole SSD replacement was unnecessary.

---

## Step 5 — If all else fails

1. **Record the EXACT error message** during the 10% failure -- screenshot Apple Configurator 2 log output. The specific error code at operation 58 tells us whether it's FTL mismatch vs. write failure vs. communication error.
2. **Check SEC EEPROM (U1950)** -- confirmed on board. If disturbed, crypto pairing is broken. U1950 had no net match at 98% coverage but is confirmed present on A2442.
3. **Thermal camera during restore attempt** -- watch for anything heating up during the 10% mark
4. **Try restore via Finder** (not Apple Configurator 2) -- Rossmann A2442 case noted DFU restore worked via Finder but not AC2
