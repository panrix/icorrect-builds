## Case history

This board was a **working BM unit** that passed both pre-QC and post-QC with flying colours (all tests passed, ammeter normal). The ONLY issue was a cracked LCD, which Andres replaced.

Then a **colour mismatch** was found (Apple says SpaceGray, BM listing says Silver). Michael told Saf to **swap this logic board into BM 1541's housing**. After the swap:

1. Charging port not working → Saf replaced it → board takes power
2. **Board always boots into Apple Diagnostics mode** instead of macOS
3. Tried DFU restore — unclear if it actually completed or errored out
4. Tried different housing — same result
5. Saf checked "NAND line fine", power button line fine, keyboard line fine
6. Saf's theory: *"customer buy this logic board from Apple and they did not configure maybe"*

## Diagnosis summary

**This is NOT a hardware fault. The ammeter reads normal (2.5–2.9A) and the board passed full QC before the swap.** The diagnostic flows (dcps_20v_triage etc.) will not help — they're for power path faults.

This is a **firmware/configuration issue** caused by one of:

1. **Stuck boot mode flag** — the logic board may have a persistent "boot to diagnostics" flag set in NVRAM/SMC. Unlikely to survive a DFU restore, but possible if DFU didn't actually complete.
2. **Keyboard sending phantom Command+D** — Apple Diagnostics is triggered by Command+D on boot. If the new housing's keyboard flex has a fault or the connector isn't fully seated, it could be sending this key combo. Saf says "keyboard line fine" but checking voltage on a line is different from confirming no phantom keypresses.
3. **macOS not installed / boot volume missing** — if DFU restore didn't complete, there's no bootable OS. What Saf calls "diagnostic mode" might actually be Internet Recovery (globe icon) or the startup manager (which looks different from Diagnostics).
4. **Incomplete Apple pairing** — Saf's theory has merit. If this was originally an Apple replacement board that wasn't fully paired via GSX/AST2, it may be stuck in a provisioning state. However, the board DID pass full QC at iCorrect before the swap, which argues against this.

## Step 1 — Clarify the symptom

**Reply to Ricky with:**

1. **What exactly appears on screen?** Describe or screenshot:
   - A grey screen with "Checking your Mac..." text and a progress bar? → That's **Apple Diagnostics**
   - A spinning globe? → That's **Internet Recovery**
   - macOS Utilities window (Disk Utility, Reinstall macOS, etc.)? → That's **Recovery Mode**
   - A setup assistant (language selection, country, WiFi)? → That's a **fresh install completing**
   - A prohibitory symbol (circle with line through it)? → **Incompatible OS**
   - A question mark folder? → **No bootable volume**
2. `Symptom: ___` (use the exact description above)

**Why this matters:** "Diagnostic mode" is ambiguous. Each of these has a completely different fix path.

## Step 2 — Confirm DFU restore status

**Reply to Ricky with:**

1. **Did the DFU restore actually complete successfully?**
   - `DFU result: COMPLETED / ERROR / DON'T REMEMBER`
   - If error: what was the error code? (4010, 4013, 4041, etc.)
2. **Which method was used?**
   - Apple Configurator 2 or Finder?
   - Was it "Revive" or "Restore"?
3. **Which USB-C port was used for DFU?**
   - On A2338, DFU port is the **left-side rear port** (closest to the hinge, left side)

## Step 3 — Rule out phantom keyboard input

Even though Saf checked "keyboard line fine", phantom keypresses are different from line voltage.

1. **Boot with keyboard flex DISCONNECTED** from the logic board. Power on via DCPS or shorting power pads.
   - `Boot without keyboard: goes to Diagnostics / goes to macOS / goes to Recovery / other: ___`
2. **If it boots normally without keyboard** → the keyboard/flex from BM 1541 housing is the problem. The board is fine.
3. **If still Diagnostics without keyboard** → move to Step 4.

---

## Step 4 — Reset NVRAM + attempt clean DFU Restore

1. **NVRAM reset** (if the board can boot far enough):
   - Power on, immediately hold **Option + Command + P + R** for 20 seconds
   - Release and let the Mac boot normally
   - `NVRAM reset: still Diagnostics / boots normally / other: ___`

2. **If NVRAM reset doesn't help → DFU Restore (full wipe):**
   - Use a **known-good USB-C to USB-C data cable** (not Thunderbolt)
   - Connect to the **left rear USB-C port** on the A2338
   - Host Mac must be on **macOS Sonoma 14+**
   - Perform a **Restore** (not Revive) — this is a full wipe
   - `DFU Restore result: SUCCESS / ERROR ___`
   - If error, screenshot the error and send to Ricky

3. **If DFU Restore completes but still boots to Diagnostics:**
   - This is abnormal. The board may have a firmware-level configuration issue.
   - ⛔ STOP — message Ricky. Possible Apple provisioning state.

---

## Step 5 — If all above fails: check if Apple pairing is needed

If:
- DFU Restore completed successfully
- NVRAM reset done
- Keyboard disconnected
- Still boots to Diagnostics

Then Saf's theory may be correct — this board was never fully paired/configured by Apple. This would require GSX/AST2 access or a visit to an Apple Authorized Service Provider.

**Before concluding this**, check one more thing:
- **Can the Mac boot from an external USB drive?** Create a bootable macOS USB, plug it in, hold Option on boot, select the external drive.
- `External boot: SUCCESS / FAILS / no option appears`
- If external boot works → the issue is specifically with the internal drive/APFS configuration, not the board itself.
