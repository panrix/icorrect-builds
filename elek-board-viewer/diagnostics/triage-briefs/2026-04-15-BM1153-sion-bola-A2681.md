## Diagnosis summary

**This is a NAND communication fault, not a power fault.**

- 20V / 1.22A sits in the A2681 flow's **0.5–1.5A (partial boot — board powering up but stalling)** scenario.
- The **question mark folder** symptom means the boot got far enough to light the display and render the folder icon → display path is working → the stall is downstream, at storage.
- **Error 4010** over DFU = host could not establish a working link to the NAND/SSD controller.
- Saf has **already replaced the NAND** and still gets the same error. That rules out the NAND chips themselves and points at the communication path: power delivery, enable signalling, or physical contact between the NAND and the SoC.

**Goal of this brief:** walk the comms path from power → enable → physical → host until we isolate which link is broken.

---

## Pre-flight (do this once before Step 0)

1. Confirm board is dry and clean under microscope — no flux residue, no solder balls on the NAND zone (near U7700 / bottom-right of the board around x≈6600–6900).
2. Confirm the replacement NAND is properly reflowed. Look for bridging on BGA edges.
3. Board on jig, 20V DCPS connected via USB-C, power off, ammeter zeroed.

---

## Step 0 — Observe the ammeter pattern

**Power on. Watch the ammeter for 30 seconds.**

| Pattern | Meaning | Action |
|---|---|---|
| **Stable ~1.22A** (doesn't move) | Stalled boot — hung at storage read. Expected for this case. | ✅ Continue to Step 1. |
| **Cycling** (1.2A → 0 → 1.2A → 0, repeating) | Boot loop — board restarting itself. Different path. | ⛔ STOP. Message Ricky. This brief does not cover cycling. |
| **Rising past 2A or staying below 0.3A** | Different scenario entirely. | ⛔ STOP. Message Ricky. |

**Why this step exists:** Ricky's Mar 25 instructions assumed stable — which is almost certainly right given the question-mark folder symptom — but we make it explicit so we don't chase the wrong branch.

## Step 1 — NAND power rails

**Measure each rail at the listed test point. Probe on the side of the board indicated — these are NOT all on the same side.**

| # | Rail name | Test point | **Side** | Expected | Ricky's original value |
|---|---|---|---|---|---|
| 1a | `PP1V25_MAME_NAND_VCC` | **CN032** | 🔻 BOTTOM | ~1.25 V | ✅ confirmed |
| 1b | `PPVDS_MAME_NAND_VCC` (likely) | **CN020** | 🔺 TOP | ~2.58 V | Ricky's value, consistent with M2 NAND DDR IO |
| 1c | NAND-SoC interface rail | **CN010** | 🔻 BOTTOM | ~0.88 V | Ricky's value — not in flow docs, trust Ricky's note |

**All three present?** → go to Step 2.

**Any rail missing (reads 0 V or shorted)?**
⛔ STOP — do NOT start tracing randomly. Message Ricky with:
- Which rail is missing
- The exact reading (0 V, half-value, short to ground?)
- A clear microscope photo of the test point you probed

The rail needs to trace back to an **MPMU buck output on schematic pages 37–43**. **The MPMU is a different chip from U8100.** U8100 is on pages 33–36 and is the SPMU / 5 V boost converter — it does NOT produce the NAND rails. Probing U8100 for a missing NAND rail wastes time. The main PMU (MPMU) is in the NAND zone of the board (right side, around x≈6400), most likely `U7700` — but Ricky will confirm the exact chip and buck number before Saf starts probing.

**Note on the flow docs:** our A2681 diagnostic flow file references `FL4000 / FL4001` fuses for the NAND rails. Those designators **do not exist on A2681** (confirmed via BRD parser — they appear to be copy-paste from another board). Don't go looking for them.

## Step 2 — NAND enable signal

**If all three power rails are present, the NAND is getting power. Now check the wake-up signal.**

| Measure | Test point | **Side** | What you're checking | Expected |
|---|---|---|---|---|
| `RP002` in diode mode | **RP002** | 🔺 TOP | Continuity on `PMU_SYS_ALIVE → NAND0_PFN_L` — the PMU's "NAND is allowed to wake up" line | Normal diode reading, not open, not shorted |

**Good reading?** → go to Step 3.

**Open or shorted at RP002?**
⛔ STOP — this is a `PMU_SYS_ALIVE` issue. Message Ricky. Do not replace RP002 blindly — we need to confirm whether the break is on the PMU side or the NAND side first.

---

## Step 3 — Microscope inspection of the NAND contact path

**With power off, under the microscope, inspect in this order:**

1. **NAND BGA pads and solder balls** — any lifted pads from the previous NAND replacement? Any bridging between adjacent balls?
2. **The NAND zone ground** — any open ground vias that got damaged during rework?
3. **The liquid damage trail** — is there corrosion extending toward the NAND or SoC beyond where it was originally cleaned? Especially on the bottom side around CN032 and CN010 (both bottom side in the NAND area).
4. **Test points CN010, CN020, CN032** — any corrosion or lifted pad on the test cap itself? Missing or partially cracked cap?

**Everything visually clean?** → go to Step 4.

**Damage found?**
⛔ STOP — photograph it. Message Ricky with the photo and the exact location before doing any rework.

---

## Step 4 — Host-side DFU retry

**Do these in order — don't skip.**

1. **Reseat the replacement NAND** if there's any doubt about reflow quality (visual reinspection first).
2. **Swap to a known-good DFU cable** — USB-C to USB-C, verified working on another board this week.
3. **Use a different host Mac** for the DFU attempt. Error 4010 can sometimes be Configurator2 / host-side quirk.
4. **Retry DFU restore** and capture the exact error code this time (screenshot of Configurator2 output).

**Error 4010 still?**
⛔ STOP — at this point the NAND communication path is physically or electrically damaged in a way that's beyond this brief. Message Ricky with all the readings from Steps 1–3 and the final Configurator2 screenshot. Likely next move is either a second NAND swap with a fresh donor or SoC-side inspection, which is a separate decision.

**DFU completes?** → ✅ board is recovered. Move to battery test / QC.

### What the ammeter jump means

**Original (Dec 2025):** 20V / 1.22A = ~24W — consistent with stalled boot at question-mark folder.
**Now (Apr 2026):** 20V / 3.2A = ~64W — this M2 MBA ships with a 30W charger. 64W at idle is physically impossible under normal operation. **There is a new short on this board that was not there in December.**

The NAND rails reading fine doesn't rule out a short — a downstream component can draw excess current while the rail voltage holds.

### Updated diagnosis

Two problems are now overlapping on this board:
1. **Original problem (Dec):** Error 4010 / NAND communication fault (the question-mark folder)
2. **New problem (Apr):** A short drawing ~2A more than before — most likely introduced during the NAND swap rework

We must resolve the short (problem 2) before we can properly diagnose whether problem 1 is still present.

## Step 6A — Short was the NANDs (current dropped)

The replacement NANDs or the reflow quality caused the excess draw. Before reinstalling, Saf must answer ALL of these:

**Reply to Ricky with:**

1. **Were the replacement NANDs blanked (erased) before installation?**
   - `NANDs were blanked: YES / NO / DON'T KNOW`
   - Non-blank NANDs cause FTL version mismatch at DFU operation 58. They MUST be erased (JCID P13 or equivalent).
2. **What model are the replacement NANDs?** Read the part number off the chip package.
   - `NAND part number: ___`
   - Cross-reference against [LogiWiki NAND List](https://logi.wiki/index.php/MacBook_NAND_List) for A2681 820-02536 compatibility.
3. **Are the NANDs in the correct positions?** They must match the original donor board positions — they are NOT interchangeable between slots.
   - `Position verified: YES / NO`
4. **Microscope photo of the NAND BGA pads** (board side, with NANDs removed):
   - Looking for: solder bridges between adjacent pads, lifted pads, missing pads, flux residue
   - Upload photo
5. **Check all small passives around the NAND footprint:**
   - Any caps knocked off, shifted, or cracked from rework heat?
   - `Nearby passives: all present / missing ___ / shifted ___`

**Do NOT reinstall NANDs until Ricky reviews the answers and photos.**

## Step 7 — Pre-DFU checklist (once short is resolved and NANDs reinstalled)

Before attempting DFU restore again, check these additional signals identified from 4010 research:

| # | Check | Test point / method | Expected | Why |
|---|---|---|---|---|
| 7a | **NAND PMIC PGOOD** (designator TBD — we cannot yet identify which IC is the NAND PMIC on A2681. BRD net extraction build is in progress to resolve this.) | See note below | ~3.7V stable | NAND power-good signal. If not asserting, SoC won't talk to NAND even though rails are present. On a Rossmann A2442 case, `NAND0_OCARINA_PGOOD` read 0.752V when it should have been 3.7V. **Note:** UP700 on A2681 is display power, NOT the NAND PMIC. Do not probe UP700 for this check. |
| 7b | `U1950` (SEC EEPROM) visual | Microscope inspection | Undisturbed | Cryptographically paired to SoC. If this chip was touched during rework, pairing is broken → restore will always fail. Cannot be replaced from donor. |
| 7c | NAND I2C lines | Measure voltage on NAND I2C bus | Normal logic levels (~1.8V or 3.3V) | On the A2442 Rossmann case, I2C lines read ~0.5V (should have been higher) — SoC could not enumerate NAND. |
| 7d | Trackpad connected | Physical check | Connected | Some M2 variants require trackpad for DFU restore to complete. Test with trackpad cable attached. |

**Reply to Ricky with all 4 readings before attempting DFU.**

## Step 8 — DFU restore (final attempt)

Only after Steps 5–7 are resolved and Ricky has approved:

1. **Board assembled with verified NANDs** (blanked, correct model, correct positions)
2. **Known-good USB-C data cable** (not Thunderbolt cable — different protocol)
3. **Host Mac running macOS Sonoma 14+**
4. **DFU port = left-most USB-C on A2681**
5. **Trackpad connected**

Attempt DFU restore. Capture:
- `DFU result: SUCCESS / ERROR ___`
- Screenshot of Apple Configurator 2 output
- If error, note at which stage it fails (e.g. "Installing system", "Updating NAND Firmware")

**If error at "Updating NAND Firmware" (operation 58):** This is the FTL version mismatch — NANDs were not properly blanked. Erase them with JCID P13 and retry.

**If error persists after all checks pass:** SoC-internal NAND controller damage. Board is BER for storage function. Data recovery not possible (AES-256 keys are fused to SoC). Discuss with Ricky whether to use for parts.
