# SOP 09.5: Shipment Confirmation to Back Market

**Version:** 1.1
**Date:** 2026-04-28
**Scope:** Notify Back Market that an accepted order has physically shipped, including tracking number and serial number.
**Owner:** `bm-shipping` webhook service (port 8013)

> **2026-04-28 change:** Lookup no longer traverses `board_relation5` from Main Board → BM Devices. The BM Sales Order ID and BM Listing ID are now read directly from the Main Board, populated by `sale-detection.js` (SOP 08). The old `board_relation5` column on Main Board points at the generic Devices catalog (3923707691), not BM Devices (3892194968), so the previous traversal was always failing the hard gate. See *Lookup* and *Boards & Columns Reference* below.

---

## Trigger

Monday webhook on Main Board (`349212843`):
- `status4` (Status) changes to **"Shipped" (index 160)**

Team physically ships the device, then changes `status4` to "Shipped" on Monday. This triggers the webhook.

**Source file:** `/home/ricky/builds/backmarket/services/bm-shipping/index.js`
**Endpoint:** `POST /webhook/bm/shipping-confirmed` → `127.0.0.1:8013` via nginx

## Flow

1. **Validate event**
   - `event.columnId === "status4"` (STATUS4_COLUMN)
   - `event.boardId === "349212843"`
   - parsed value index === 160 (`Shipped`)

2. **Identify the exact Main Board item**
   - Part B does not search Monday by name, buyer, or listing ID
   - it starts from the Monday webhook event item itself:
     - `event.itemId`
   - this item on Main Board is the source of truth for shipment confirmation

3. **Dedup check**
   - query last 5 Monday updates
   - if any contain `BM notified`, skip as already processed

4. **Fetch tracking, serial, BM order ID, BM listing ID from Main Board** (single query)
   ```graphql
   { items(ids: [ITEM_ID]) {
     column_values(ids: ["text53", "text4", "text_mm2vf3nk", "text_mm2v7ysq"]) {
       id text
     }
   } }
   ```

   | Column ID | Title | Purpose |
   |-----------|-------|---------|
   | `text53` | Outbound Tracking | Tracking number (strip spaces) |
   | `text4` | IMEI/SN | Serial number |
   | `text_mm2vf3nk` | BM Sales Order ID | Order ID for BM API call (written by SOP 08) |
   | `text_mm2v7ysq` | BM Listing ID | Used to resolve the BM Devices entry for post-success cleanup |

5. **HARD GATE: No tracking number**
   - Slack alert: `⚠️ {itemName} marked Shipped but no tracking number found. Manual BM update needed.`
   - return without notifying BM

6. **HARD GATE: No serial number**
   - Slack + Telegram alert with tracking number
   - return without notifying BM

7. **HARD GATE: No BM Sales Order ID**
   - Slack alert: `⚠️ {itemName} has tracking but no BM Sales Order ID on the Main Board. Backfill or check sale-detection.js.`
   - return without notifying BM

8. **Resolve BM Devices entry by BM Listing ID** (best-effort; cleanup-only)
   ```graphql
   { items_page_by_column_values(board_id: 3892194968, columns: [{column_id: "text_mkyd4bx3", column_values: ["{BM_LISTING_ID}"]}], limit: 5) { items { id } } }
   ```
   - 1 match → use for post-success cleanup writes (group move + clear sale fields).
   - 0 or >1 matches → still proceed with the BM API call. Skip cleanup, raise a Slack warning so ops can finish manually.

10. **Notify BM of shipment**
   ```
   POST https://www.backmarket.co.uk/ws/orders/{bmOrderId}
   Headers:
     Authorization: $BACKMARKET_API_AUTH
     Content-Type: application/json
   Body: {
     "order_id": "{bmOrderId}",
     "new_state": 3,
     "tracking_number": "{tracking_number}",
     "tracking_url": "https://www.royalmail.com/track-your-item#/tracking-results/{tracking_number}",
     "shipper": "Royal Mail Express",
     "serial_number": "{text4 serial}"
   }
   ```

   Notes:
   - tracking number must have spaces stripped before sending
   - serial number is a hard gate in the standalone `bm-shipping` service

11. **After success**
   - Slack + Telegram: `✅ BM notified of shipment: {itemName} - Order {bmOrderId} - Tracking {tracking}`
   - Monday comment:
     ```
     ✅ BM notified of shipment [{timestamp}]
     Tracking: {tracking}
     Order: {bmOrderId}
     ```
   - BM order state transitions: state 3 (our submission) → state 9 (shipped/complete, BM sets this)

12. **Move BM Devices item to Shipped group**
   ```graphql
   mutation { move_item_to_group(
     item_id: BM_DEVICE_ITEM_ID,
     group_id: "new_group269"
   ) { id } }
   ```

   | Group | ID | Board |
   |-------|----|-------|
   | Shipped | `new_group269` | BM Devices (3892194968) |

## Error Handling

### BM API Errors

| HTTP Status | Action |
|-------------|--------|
| 2xx | Success: post Slack confirmation + Monday comment |
| 5xx | Retry ONCE after 30 seconds. If retry fails: Slack alert `❌ BM update FAILED`. Manual update needed. |
| 4xx | Do NOT retry. Slack alert with error. Likely causes: order already shipped, order cancelled, invalid tracking. |

### Common Failure Modes

| Failure | Detection | Action |
|---------|-----------|--------|
| No tracking on Main Board | `text53` empty | Webhook alerts Slack, does not notify BM |
| No serial on Main Board | `text4` empty | Webhook alerts Slack + Telegram, does not notify BM |
| No BM Sales Order ID on Main Board | `text_mm2vf3nk` empty | Webhook alerts Slack with tracking; backfill or check sale-detection.js |
| BM Devices cleanup ambiguous | listing_id 0/many matches | BM is still notified; Slack warns that BM Devices entry needs manual move/clear |
| BM already notified | `BM notified` in recent Monday updates | Skip (dedup) |
| Wrong tracking format | Spaces in tracking number | Code strips spaces automatically |

- This SOP does NOT handle returns (that is SOP 12)

## Change log — 2026-04-28

- Removed `board_relation5` traversal. Investigation confirmed `board_relation5` on Main Board is configured to link to board `3923707691` (generic Devices catalog), not BM Devices. Every BM ship-confirmation since the service was created (2026-03-26) had been silently failing the "no BM order ID" hard gate.
- Added two text columns on Main Board: `text_mm2vf3nk` (BM Sales Order ID) and `text_mm2v7ysq` (BM Listing ID).
- `sale-detection.js` (SOP 08) writes these on the same mutation that sets status → Sold and date sold.
- `bm-shipping` reads them directly from the Main Board item resolved from the webhook `itemId`. No traversal.
- Post-success BM Devices cleanup now resolves the entry via `items_page_by_column_values` on listing ID. Treated as best-effort: if 0 or >1 matches, BM is still notified and Slack warns ops to finish the BM Devices side manually.
- Existing items predating this change need a one-shot backfill: `node /home/ricky/builds/backmarket/scripts/backfill-bm-order-ids.js [--dry-run]`.
