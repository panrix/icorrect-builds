# SOP 6.5: Listings Reconciliation

**Version:** 1.0
**Date:** 2026-03-22
**Scope:** Daily reconciliation between Monday boards and Back Market active listings. Ensures data integrity before buy box management (SOP 7) runs.
**Owner:** Backmarket agent (Hugo) / Jarvis

---

## Trigger

- Daily cron, runs AFTER V6 scraper and BEFORE buy box check (SOP 7)
- On-demand when requested

**Daily run order:**
1. V6 sell price scraper
2. **SOP 6.5: Listings reconciliation** ← this SOP
3. SOP 7: Buy box management

---

## Step 1: Gather data from both sources

### 1a: Monday — all "Listed" items
Query Main Board (349212843), group "BMs Awaiting Sale" (`new_group88387__1`), filter `status24` = Listed (index 7).

For each item, read:
- Item ID, name
- `status24` (Repair Type / listing status)

### 1b: BM — all listings
`GET /ws/listings` paginated. For each listing:
- `listing_id`, `quantity`, `grade`, `title`, `price`, `sku`, `product_id`

### 1c: BM Devices Board — linking data
For each Monday "Listed" item, find the corresponding BM Devices Board (3892194968) entry via board_relation (`... on BoardRelationValue { linked_item_ids }`).

Read:
- `text_mkyd4bx3` (stored listing ID)
- `numeric_mm1mgcgn` (Total Fixed Cost)
- `numeric` (Purchase Price)

---

## Step 2: Cross-reference and identify mismatches

### Check A: Monday Listed → BM Active
For each Monday "Listed" item:
1. Get its listing ID from BM Devices `text_mkyd4bx3` (single source of truth)
2. Find that listing_id in BM active listings
3. If found and qty > 0: ✅ matched
4. If found but qty = 0: ⚠️ Listed on Monday but offline on BM
5. If not found: ⛔ Listed on Monday but no matching BM listing

### Check B: BM Active → Monday Listed
For each BM active listing (qty > 0):
1. Search BM Devices Board for items with `text_mkyd4bx3` matching the listing_id
2. Find the linked Main Board item via `board_relation`
3. Check Main Board `status24`:
   - If "Listed" (7): ✅ matched
   - If "Sold" (10): ⛔ **OVERSELL RISK** — device sold but still live on BM
   - If "To List" (8): ⚠️ BM active but Monday not updated
   - If any other status: ⛔ **OVERSELL RISK** — device not in listable state
   - If no Monday item found: ⛔ **ORPHAN LISTING** — active on BM with no Monday tracking

### Check C: Quantity reconciliation
For each product_id + grade combination:
1. Count Monday "Listed" items that map to this listing
2. Compare to BM listing qty
3. If Monday count > BM qty: ⚠️ missed revenue (more devices than listed)
4. If BM qty > Monday count: ⛔ **OVERSELL RISK** (listing shows more than we have)

### Check D: Missing BM Devices entry
For each Monday "Listed" item:
1. Check if a BM Devices Board entry exists (via board_relation)
2. If no entry: ⛔ **MISSING BM DEVICE** — cannot track listing ID, costs, or profitability

### Check E: Missing cost data
For each BM Devices entry linked to an active listing:
1. Check `numeric_mm1mgcgn` (Total Fixed Cost) is present and > 0
2. If missing: ⚠️ flag for backfill
3. If Main Board link exists: auto-calculate from purchase + parts + labour + shipping + buy fee
4. If no Main Board link: flag for manual entry

## Step 3: Automated actions (safety only)

**Auto-take offline (no approval needed):**
- BM listing active but Monday status = "Sold", "Shipped", "BM RTN/RFD", or any non-Listed status
- BM qty > Monday device count for that spec

**Auto-fix (no approval needed):**
- Missing Total Fixed Cost where data can be calculated from Monday columns

**Flag only (needs review):**
- Monday says Listed but BM listing is offline/missing
- Monday count > BM qty (missed revenue)
- Missing BM Devices entry
- Orphan BM listings (no Monday match)

## Step 5: Missing BM Device Protocol

When a Monday "Listed" item has no BM Devices Board entry:

1. **Create BM Devices item** with:
   - Name: same as Main Board item name
   - `board_relation`: link to Main Board item
   - `text` (model): extract from Main Board
   - `numeric` (purchase price): from Main Board
2. **Read specs from Main Board** and populate BM Devices columns
3. **Flag to BM Telegram**: "Created BM Devices entry for {device}. Listing ID and costs need populating."
4. **Do NOT auto-list**. The device needs to go through SOP 6 properly.

## Columns used

### Main Board (349212843)
| Column ID | Purpose |
|-----------|---------|
| `status24` | Listing status (Listed=7, To List=8, Sold=10) |

> Listing ID is NOT on the Main Board. Single source of truth is BM Devices `text_mkyd4bx3`.

### BM Devices Board (3892194968)
| Column ID | Purpose |
|-----------|---------|
| `text_mkyd4bx3` | Listing ID |
| `numeric_mm1mgcgn` | Total Fixed Cost |
| `numeric` | Purchase Price |
| `board_relation` | Link to Main Board (use `... on BoardRelationValue { linked_item_ids }`) |

---

## Error handling

- Monday API failure: abort reconciliation, alert. Do not run SOP 7 without reconciliation.
- BM API failure: abort, alert.
- If reconciliation finds oversell risks: resolve those BEFORE running SOP 7.

---

## What does NOT happen

- No relisting without going through SOP 6
- No cost guessing — either calculate from real data or flag
- No qty changes without physical verification

## Future additions (noted Mar 22)

### Remediation steps
Each issue type needs defined next steps. Currently flagged but remediation is manual. Need to document:
- Monday Listed / BM offline → check why, relist via SOP 06 or update Monday status
- No listing ID → set status to To List, SOP 06 picks up
- Missing BM Device → create entry (Step 5), set to To List
- Orphan listing → investigate, link to Monday or take offline

### BM Devices board group verification
Script should verify that active BM Device items are in the "BM To List / Listed / Sold" group (`new_group`). Items in wrong groups (e.g. "BM Returns") should be flagged. A daily cron to check group placement would catch items that get stuck after returns are processed.
