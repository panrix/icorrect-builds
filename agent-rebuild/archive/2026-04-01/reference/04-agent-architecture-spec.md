## Design Principles

1. **Structure over engineering.** The right files in the right folders solves 80% of memory/context problems. Build infrastructure only when file-based organisation outgrows itself.
2. **Agents are the last step, not the entire pipeline.** Raw data is pre-processed by cron jobs before agents see it. Agents receive structured, clean markdown.
3. **Readonly by default.** Agents read data and produce recommendations/responses. They do not modify source systems directly unless explicitly given that skill.
4. **SOPs are present, not discovered.** SOPs live in `sops/` and are indexed by memory-core with `extraPaths`. The agent doesn't decide whether to read an SOP -- it's part of the searchable workspace.
5. **One agent, one domain.** No agent handles multiple business domains. If a domain has sub-workflows (BM: trade-in, sales, returns), the same agent handles all with per-workflow SOPs.
6. **Memory is captured automatically.** Agents write markdown naturally. session-memory hook saves conversations. Memory flush catches pre-compaction. Zero agent compliance required.
7. **Workspaces are controlled.** Defined file structure. Anything outside it gets flagged. No clutter.
8. **Start small, grow organically.** 4 agents at launch. Sub-agents added only when data proves the need and an SOP exists for what the sub-agent would do.

## Agent Template: Workspace Structure

Every agent workspace follows this structure. Nothing else at root.

```
~/.openclaw/agents/{agent-id}/workspace/
  SOUL.md              # Identity and purpose (max 40 lines)
  CLAUDE.md            # Operating rules (max 80 lines)
  USER.md              # Who Ricky is, how he works

  sops/                # Domain-specific SOPs
    index.md           # SOP list with trigger conditions
    SOP-{name}.md      # Individual SOPs (standalone)

  data/                # Cron-refreshed, agent reads only
    {feed-name}.md     # Pre-processed data feeds (structured markdown)

  decisions/           # Structured decision log
    {topic}.md         # What was decided, when, why, alternatives considered

  memory/              # Agent writes here naturally + session-memory hook auto-saves
    YYYY-MM-DD-*.md    # Session logs (auto-generated by session-memory hook)
    {topic}.md         # Agent-written observations and learnings

  foundation/          # Symlink to ~/.openclaw/shared/ (readonly)
```

### What is NOT allowed in workspace root:
- No legacy files (*-legacy.md)
- No transcript dumps (file_4*.json, *.srt, *.vtt)
- No project repos (shopify themes, scraped HTML, etc.)
- No stale TODO.md, WORKING.md, battle-plan.md
- No docs/ directory (replaced by sops/ and data/)
- No scripts (*.py, *.sh) -- scripts live outside agent workspaces
- No JSON/YAML data dumps

### Filesystem Permissions (Hard Enforcement)

Instructions alone don't work — v2 proved agents ignore CLAUDE.md rules. Filesystem permissions are the primary enforcement mechanism.

After workspace creation, set ownership and permissions so the agent process can only write where intended:

| Directory | Agent access | Managed by | How |
|-----------|-------------|------------|-----|
| Workspace root (*.md) | Read-only | Code / setup script | `chmod 444` on SOUL.md, CLAUDE.md, USER.md |
| `sops/` | Read-only | Code + Ricky (manual) | Owned by `ricky`, agent cannot write |
| `data/` | Read-only | Cron jobs | Owned by `ricky`, agent cannot write |
| `decisions/` | Read-only | Nightly cron | Owned by `ricky`, agent cannot write |
| `foundation/` | Read-only | Symlink to shared/ | Already chmod 444 |
| `memory/` | **Read-write** | Agent + session-memory hook | Agent's only writable directory (plus research/findings/ during research phase) |
| `research/findings/` | **Read-write** | Agent (research phase only) | Removed after transition to department head |

**Implementation:** During workspace setup, create all directories owned by `ricky:ricky`. Create a dedicated group or use directory permissions so the OpenClaw gateway process can write to `memory/` and `research/findings/` only. All other directories are agent-readable via the workspace but not writable.

**Note:** OpenClaw runs as the `ricky` user, so true OS-level user separation isn't possible without containerisation. The practical enforcement is:
- Root files (SOUL.md, CLAUDE.md, USER.md) set to `chmod 444` — agent tools will fail to write
- `data/`, `sops/`, `decisions/` files set to `chmod 444` after cron writes them
- CLAUDE.md explicitly states only `memory/` and `research/findings/` are writable
- Watchdog cron (below) catches anything that slips through

This is defence in depth: instructions + chmod + watchdog. No single layer is relied upon alone.

### Workspace Watchdog Cron

A weekly cron scans all agent workspaces and alerts Ricky if anything exists outside the defined structure. Does not auto-delete — alerts only.

**What counts as a violation:**
- Any file in workspace root that is not SOUL.md, CLAUDE.md, USER.md, or MEMORY.md
- Any directory in workspace root that is not sops/, data/, decisions/, memory/, foundation/, or research/
- Any non-markdown file anywhere in the workspace (e.g., .json, .py, .sh, .csv, .srt, .vtt)
- Any file in `data/` or `sops/` not owned by ricky (i.e., written by the agent, not by a cron)

**What it does NOT flag:**
- Files inside `memory/` (agent is allowed to write freely here)
- Files inside `research/findings/` (during research phase)
- The foundation/ symlink itself

**Alert format (posted to Agent Activity Telegram group):**
```
⚠️ Workspace Watchdog — {date}

{agent-id}: 2 violations
  - ROOT: TODO.md (unexpected file in root)
  - data/raw-dump.json (non-markdown file in data/)

{agent-id}: clean ✓
```

**Schedule:** Weekly, Sunday 11pm UTC (before the weekly summary cron).
**Posts to:** Agent Activity group (-1003896233105).
**Script location:** `/home/ricky/scripts/workspace-watchdog.sh` (built during Phase 2).

## SOP Structure

Each SOP is a standalone markdown file in `sops/`.

```markdown
# SOP: {Name}
**Domain:** {agent-id}
**Trigger:** {What type of message/situation triggers this SOP}
**Last verified:** {Date}
**Verified by:** {Who confirmed this matches reality}

## Prerequisites
- {What must be true before starting this flow}

## Steps
1. {Action} -- {Why}
2. {Action} -- {Why}
3. {Decision point: if X then go to step 4, if Y then go to step 6}
4. ...

## Common Errors
- {Error}: {What to do}

## Escalation
- If {condition}: escalate to {who} via {channel}
```

### SOP Index (sops/index.md)

```markdown
# SOPs -- {Agent Name}

| SOP | Trigger | File |
|-----|---------|------|
| Daily operations | Heartbeat / start of day | SOP-daily.md |
| Trade-in processing | Message contains trade-in order | SOP-trade-in.md |
| Sales listing | Message about creating/updating listings | SOP-sales.md |
| Return handling | Message about return/dispute | SOP-return.md |
```

## How Context Works (The Memory Solution)

### The principle: right files, right folders, right loading

The previous approach tried to solve memory with engineering: embedding pipelines, custom plugins, semantic retrieval, auto-injection hooks. Research (doc 05) showed this was premature. The actual problem is structural -- workspaces are cluttered, data is unstructured, and everything gets dumped into context wholesale.

### How OpenClaw loads agent context (already built-in):

| Mechanism | What it does | When |
|-----------|-------------|------|
| Bootstrap | Injects SOUL.md, CLAUDE.md, USER.md, MEMORY.md at session start | Every session |
| session-memory hook | Saves last 15 conversation lines to `memory/YYYY-MM-DD-slug.md` on `/new` | On new session |
| Memory flush | Before compaction, prompts agent to save durable notes | Before compaction |
| memory-core | Indexes all `.md` files in `memory/` (+ extraPaths) with hybrid search | Ongoing |
| memory_search tool | Agent can search indexed memory when it needs deeper context | On demand |

### What we configure (not build):

```json
{
  "agents": {
    "defaults": {
      "memorySearch": {
        "extraPaths": ["sops/", "data/", "decisions/"],
        "query": {
          "maxResults": 6,
          "minScore": 0.35,
          "hybrid": {
            "enabled": true,
            "vectorWeight": 0.7,
            "textWeight": 0.3,
            "temporalDecay": {
              "enabled": true,
              "halfLifeDays": 30
            }
          }
        }
      }
    }
  }
}
```

This means memory-core indexes everything in `memory/`, `sops/`, `data/`, and `decisions/`. The agent can search across all of it. SOPs, current data, past decisions, and conversation history are all searchable.

### What happens in practice:

1. Agent boots -> SOUL.md, CLAUDE.md, USER.md loaded (bootstrap, built-in)
2. Agent gets a message about a trade-in return
3. CLAUDE.md tells it: "Your SOPs are in sops/. Read sops/index.md to see what is available."
4. Agent reads sops/index.md, finds SOP-return.md, reads and follows it
5. Agent checks data/active-returns.md for current return status
6. Agent writes observations to memory/returns-handling.md
7. Next session: memory-core has indexed the observation. memory_search can find it.

No embedding pipeline to build. No custom plugin. No auto-injection hook. Just files in folders, indexed by what's already there.

### When we add auto-injection (future, not now):

If and when memory volume outgrows file-based organisation (hundreds of decisions, thousands of facts), we have a proven path:
- `before_prompt_build` hook fires every message, receives user prompt + session history
- Can query memory-core's search engine and inject top-K results as `prependContext`
- This is deferred to a future phase. The infrastructure is understood and documented in 05.

## Decision Archival

A nightly cron processes `memory/*.md` files and extracts decisions into `decisions/{topic}.md`.

### Decision format:
```markdown
## YYYY-MM-DD: {Decision summary}
- **Decision:** {What was decided}
- **Reasoning:** {Why}
- **Alternatives considered:** {What else was discussed}
- **Decided by:** {Who}
```

Uses Haiku for extraction (cheap, fast) or regex for structured patterns. Runs nightly. Agents read `decisions/` for past context.

## Supabase as Source of Truth

Supabase and agent workspaces serve different purposes:

| Layer | What | Who reads |
|-------|------|-----------|
| **Supabase** | Central business data store. Aggregates from Monday, BM, Xero, Intercom. | Crons, dashboards, APIs |
| **Agent data/** | Markdown snapshots of Supabase/API data for agent consumption. | Agents (readonly) |
| **Agent memory/** | Conversation history, observations, learnings. | Agents (read/write) |
| **Agent decisions/** | Structured decision log extracted from memory. | Agents (read), cron (write) |

The relationship: **Supabase is the database. Agent workspaces are the view layer.** Crons are the bridge -- they query Supabase (or source APIs), format as markdown, and write to agent `data/` folders.

A Supabase audit (Phase 1.5) is needed to understand what data is available, what is missing, and whether crons should pull from Supabase or directly from source APIs.

## Skills (After SOPs Are Verified)

Skills make SOPs executable. They are NOT written during the research phase.

**Order:** SOP verified with Ricky -> Code verifies technical accuracy -> Skill written based on SOP -> Skill tested -> Skill deployed to production agent.

Research agents do not need Skills. Their job is conversation and documentation.
Production agents get Skills after their SOPs are verified and the workspace is built.

Each Skill maps to one SOP. The SOP is the spec. The Skill is the implementation.

## Success Criteria

How we know v3 is working:

1. **BackMarket agent follows SOPs** -- because they are in its workspace and CLAUDE.md tells it to use them
2. **Zero workspace clutter** after 2 weeks (weekly cron confirms)
3. **Memory persists across sessions** -- session-memory hook auto-saves, memory-core indexes
4. **Data feeds are current** -- cron health checks confirm feeds update on schedule
5. **Ricky can add a new agent in < 1 hour** by copying template and writing SOPs
6. **Context switching works** -- agent handles team -> SOPs -> back to team without losing the thread (because decisions/ and memory/ retain structured context)
