# BUILD BRIEF: Alex Triage Classifier Refactor

## Problem

The triage system has two fundamental bugs:

1. **No differentiation** — every single card shows "🟡 Needs review" with "Not in catalogue — diagnostic needed" regardless of context. Yesterday's run produced 47 identical-quality cards.

2. **Pricing always fails** — `findPrice(index, deviceModel, repairType)` is called with `repairType=null` every time, so it always returns null. Even for "OLED screen replacement for an iPhone XS Max", the system doesn't look up screen replacement pricing.

## What Must Change

### 1. Five-Way Classification (replace current `classifyConversation`)

Current categories: `new_enquiry`, `follow_up`, `complaint`, `chase`, `bm_email`

New categories:

**A. `active_repair`** — Active repair already on Monday
- Has a Monday match with status NOT in ["Returned", "Shipped", "Cancelled/Declined", "BER/Parts"]
- Customer is chasing status, asking about timeline, or confirming parts/details
- Key: Monday data exists, status is in progress. The answer is not pricing — it's status.
- Confidence: can be 🟢 green if status is clear.

**B. `new_enquiry`** — New customer asking for a repair price
- No Monday match OR Monday match confidence < 0.5
- Customer says "how much is X", "can you fix Y", "I need a quote"
- Needs a price. If in Shopify catalogue, show it. If not, suggest diagnostics.
- Confidence: always 🟡 yellow (needs review before send).

**C. `quote_followup`** — Quote sent, customer hasn't decided
- There's a Monday match with a quote amount already set
- Customer is hesitating, asking about alternatives, mentioning price sensitivity
- Price exists already. Needs commercial judgement.
- Confidence: 🟡 yellow (needs human review on pricing flexibility).

**D. `complaint_warranty`** — Something went wrong
- Keywords: complaint, unhappy, refund, not acceptable, disgusted, trading standards, warranty claim, "still not working", "came back broken"
- Or Monday shows a warranty return / recurring repair
- Always 🟡 red escalation tier (no approve button, only escalate/snooze).
- No pricing displayed.

**E. `corporate_account`** — Business relationship, not a single repair
- Sender email is a business/organisational domain (not Gmail/Outlook/etc)
- OR Monday item shows client_status containing "Corporate" or "Client"
- OR conversation mentions procurement, purchase order, company billing, staff devices
- Always 🟡 yellow but flagged for special handling and visible corporate tag.

**F. `bm_email`** — Back Market related (keep existing)
- Email from/about BackMarket
- Always route with special tag

### Classification Decision Tree

```
1. Is email from BackMarket? → bm_email
2. Does the conversation mention complaint/unhappy/refund/not acceptable OR is Monday showing repeated warranty returns? → complaint_warranty
3. Is Monday match confidence >= 0.7 with an ACTIVE status (not terminal) AND is the customer asking about their specific repair? → active_repair
4. Is Monday match confidence >= 0.7 AND has a quote_amount AND customer mentions price hesitation or asks about alternatives? → quote_followup
5. Is sender a business domain OR does Monday show corporate client_status? → corporate_account
6. Otherwise → new_enquiry
```

### 2. Repair Type Extraction for Pricing

**Problem:** `findPrice()` is called with `repairType=null` for all new enquiries.

**Fix:** Extract repair type from the customer's initial message text before looking up pricing.

**Implementation** — add `extractRepairType(messages)` to `lib/triage.js`:
- Scan the first customer message (or source body) for repair type keywords
- Match against known Shopify product catalogue patterns:
  - `screen replacement`, `LCD`, `cracked glass`, `screen`, `display`
  - `charging port`, `charge port`, `USB-C port`, `lightning port`
  - `back glass`, `rear glass`, `back cover`
  - `battery` (but not "battery draining" or software context)
  - `headphone jack`
  - `power button`, `volume button`, `mute button`
  - `front camera`, `back camera`, `rear camera`, `camera`
  - `speaker`, `earpiece speaker`, `bottom speaker`
  - `microphone`, `mic`
  - `trackpad`
  - `keyboard`
  - `face id`, `touch id`
  - `SIM tray`, `SIM reader`
  - `water damage`, `liquid damage`
  - `dead`, `won't turn on`, `no power`
- Return the best match as a normalized label that matches Shopify product naming, or null if nothing matches clearly.

### 3. Wire Device + Repair Type to Pricing in `inbox-triage.js`

Current flow:
```
price = findPrice(pricingIndex, mondayMatch?.device_model, mondayMatch?.repair_type)
```
This only works when Monday has both fields, which is rare.

New flow:
```
// Primary: use Monday data if available
let price = findPrice(pricingIndex, mondayMatch?.device_model, mondayMatch?.repair_type)

// Fallback: extract from conversation text for new enquiries
if (!price) {
  const detectedDevice = detectDeviceFromMessages(messages)
  const detectedRepair = extractRepairType(messages)
  if (detectedDevice && detectedRepair) {
    price = findPrice(pricingIndex, detectedDevice, detectedRepair)
  }
}
```

The `detectDeviceFromMessages()` function already exists in `lib/triage.js` — it just needs to be used before calling `findPrice`.

### 4. Confidence Tier Updates

Current logic: green if price is known AND Monday match >= 0.85. Yellow always for new enquiries.

Keep this design but update the `what_matters` summary per category:

- `active_repair`: "Existing repair in progress. Monday shows [status]. Draft a status update."
- `new_enquiry` with price: "New enquiry, price available: £[price]. Draft a reply with pricing and next steps."
- `new_enquiry` no price: "New enquiry, no catalogue match. Suggest walk-in diagnostics."
- `quote_followup`: "Quote of £[amount] sent. Customer is [hesitating/asking questions]. Needs commercial judgement."
- `complaint_warranty`: "[Complaint/warranty] — needs careful wording and Ferrari visibility."
- `corporate_account`: "Corporate account [company name]. Escalate for relationship management."
- `bm_email`: "Back Market related email. Route with priority."

### 5. Keyboard Button Adjustments

- `active_repair`: If status is clear and unambiguous, show Approve button (confidence can be green)
- `new_enquiry`: Always show Approve/Edit but never green confidence (always yellow for review)
- `quote_followup`: No Approve button — Edit/Only. This needs human pricing judgement
- `complaint_warranty`: Red tier — only Escalate/Snooze, no Approve
- `corporate_account`: Yellow — Edit/Escalate options visible
